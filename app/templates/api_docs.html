API reference

Point your scripts — or your AI — at Warpyard. Everything the dashboard does, you can do over HTTP.
API reference Platform handbook
Base URL
{{ settings.PUBLIC_URL }}/api/v1
Auth
Send an API key as a bearer token:
Authorization: Bearer wy_your_key_here

Create and revoke keys on your Account page. Keys are shown once.

Responses
JSON. Errors return {"detail": "…"} with a 4xx status. Actions that change a server return 202 Accepted — the work runs asynchronously; poll the server to watch it settle.

Quick start — create a server

curl -s {{ settings.PUBLIC_URL }}/api/v1/plans \
  -H "Authorization: Bearer $WY_KEY"

curl -s -X POST {{ settings.PUBLIC_URL }}/api/v1/servers \
  -H "Authorization: Bearer $WY_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name":"my-app","plan":"wy-1-1","image":"ubuntu-24.04"}'

To give an AI (or yourself) SSH access, add a key before creating the server — keys are installed at build time.

Servers

GET
/servers
List your live servers.
GET
/servers/{id}
One server — status, URL, SSH, domains, deploy remote.
POST
/servers
{name, plan, image} — create a server.
POST
/servers/{id}/actions/{action}
action ∈ start · stop · reboot · rebuild · destroy.
POST
/servers/{id}/resize
{plan} — move to a bigger plan (disk grows only).

Backups — nightly off-host backups, kept 7 daily + 4 weekly

GET
/servers/{id}/backups
List a server's restore points.
PUT
/servers/{id}/backups
{enabled} — turn nightly backups on or off.
POST
/servers/{id}/backups/actions/backup
Take a backup now (server keeps running).
POST
/servers/{id}/backups/actions/restore
{backup_id} — replace the server's disk with a restore point. Destructive; the server reboots.

Snapshots — instant on-host restore points

GET
/servers/{id}/snapshots
List a server's snapshots.
POST
/servers/{id}/snapshots
{name} — take a snapshot now (synchronous).
POST
/servers/{id}/snapshots/{name}/actions/rollback
Roll back to a snapshot. Destructive; the server reboots.
DEL
/servers/{id}/snapshots/{name}
Delete a snapshot.

Monitoring & metrics

PUT
/servers/{id}/monitoring
{enabled} — uptime checks + email alerts (a PoppaPing account is set up for your email if you haven't connected one).
GET
/servers/{id}/monitoring?period=24h
Uptime % and response-time history (24h · 7d · 30d · 90d).
GET
/servers/{id}/metrics?timeframe=hour
CPU / memory / network history from the host (hour · day · week).

Settings

PUT
/servers/{id}/restart-schedule
{enabled, at} — nightly reboot at "HH:MM" UTC. Game servers wait until nobody's playing.
PUT
/servers/{id}/share
{enabled, note} — list the server on the members-only board while it's running.
PUT
/account/private-network
{enabled} — let your own servers reach each other on the private network.

Push to deploy

Any OS server (Ubuntu / Debian) can become a git push target — Netlify-style, but the build & serve happen on your own box (Caddy + Node). No control-plane builds. 

# 1. one time, from your computer — installs the add-on on the server over SSH:
ssh -p <ssh_port> root@{{ settings.EDGE_HOST }} "curl -fsSL {{ settings.PUBLIC_URL }}/setup.sh | bash -s pushdeploy"

# 2. one time, in your project:
git remote add warpyard ssh://root@{{ settings.EDGE_HOST }}:<ssh_port>/srv/site.git

# 3. every deploy after that:
git push warpyard main

All three commands — with your server's SSH port already filled in — are on the server's page under Add‑ons. Push main; if the repo has a package.json it runs npm ci && npm run build and serves dist/ (or build/out/public), otherwise the files as-is.

Prefer an AI-native connection?

Warpyard is also a Model Context Protocol server — add it to Claude or Cursor and your AI can manage servers directly (OAuth sign-in, no API key to paste).
