What Warpyard is

Warpyard is a small, invite-only VM hosting platform. Every server you create is a real KVM virtual machine on dedicated hardware — not a container, not a shared runtime. You get root, your SSH keys, a public hostname with automatic HTTPS, and a browser console. A control plane handles provisioning and lifecycle; a public edge handles ingress. Everything the dashboard does is also available over the REST API and MCP.

Images come in three kinds: plain OS (Ubuntu 24.04, Debian 12), apps (WordPress, Nextcloud, Gitea, …) that come up configured and serving, and game servers (Minecraft, Valheim, CS, …) with their ports pre-published. Any OS server can add push‑to‑deploy or Docker with one command — see its Add‑ons tab (push‑to‑deploy = git push and the site redeploys).

Lifecycle semantics worth knowing: create usually lands in well under a minute; resize is grow-only and power-cycles the server; rebuild and destroy wipe the disk and cannot be undone. Anything that changes a server is asynchronous — the API returns immediately and the server settles through provisioning → booting → running.

Networking & ingress

Your VM has no public IP. It lives on an isolated tenant network, and the public edge publishes exactly what should be reachable:

Web
https://<name>.{{ settings.BASE_DOMAIN }} → your server's port 80. Custom domains can be added on top; HTTPS is automatic (Let's Encrypt) once DNS points at the edge.
SSH
ssh -p <port> root@{{ settings.EDGE_HOST }} — each server gets a stable forwarded port (shown on its Access tab).
Game ports
Ports a game image declares (e.g. Minecraft 25565) are published on the edge with stable public ports.

Outbound internet from the VM is open. Certificates are real Let's Encrypt issuances — the platform shares LE's rate limits, so churning many new hostnames in one week can briefly defer new certs.

Security model

Tenant isolation
Every VM carries a platform-managed firewall: default-deny inbound, and its source IP + MAC are pinned (anti-spoofing). Servers cannot reach each other — not yours, not anyone's — except through the public edge like any other internet client.
Private networking
Opt-in, per account: your own servers (and only yours) may talk to each other directly. Cross-account traffic is never possible regardless of settings.
End-to-end TLS
On OS and app images (default), the VM holds its own certificate and terminates its own HTTPS — the edge forwards ciphertext and never sees plaintext traffic.
Disk encryption
Optional at creation: the disk is stored encrypted at rest (AES-256). Costs about a minute of extra build time.
Access
Only your SSH keys are installed on your servers — the platform keeps no standing login to your VM. The browser console is gated to the server's owner. API keys are stored hashed and shown once; passwords are bcrypt.

Restoring either one replaces the server's current disk — that part is destructive by nature. Backups run while the server stays up. For anything you'd miss: snapshots before risky work, backups on always.

Performance expectations

VMs get full host CPU passthrough (Xeon E5-2690 v4 — AES-NI, AVX2, the works), so software that expects a modern CPU behaves normally and TLS is hardware-accelerated. vCPUs are shared, not dedicated: bursty and steady web/game workloads are the design target; sustained all-core batch crunching will feel the neighbors (and they'll feel you). Storage is ZFS-backed, which is what makes snapshots instant. Network is rate-limited per plan (table above).

Optional per-server nightly restart is player-aware on game servers — an occupied server waits until it's empty. Uptime monitoring with email alerts is one click on the server's Monitoring tab.

Limits & housekeeping

Accounts start with a 15-server limit (ask if you genuinely need more). Members get a couple of invite slots for friends. Sharing a server on the Board lists it for other members while it's running — nothing on the board is visible outside the membership.
